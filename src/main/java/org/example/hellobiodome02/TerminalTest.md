## 💻 실행 환경 설정

`java_first_course` 디렉토리에서 아래 명령어를 실행한다.

**1단계 — 출력 디렉토리 생성 및 컴파일**
```bash
# Windows (PowerShell / CMD)
mkdir out
javac -d out src\main\java\org\example\hellobiodome02\HelloBiodome02.java

# macOS / Linux
mkdir out
javac -d out src/main/java/org/example/hellobiodome02/HelloBiodome02.java
```

**2단계 — 실행 기본 형태**
```bash
java -cp out org.example.hellobiodome02.HelloBiodome02 <태양광> <풍력> <지열>
```

## 🎯 테스트 케이스

편의를 위해 이하 커맨드에서 클래스패스 부분은 `[CP]`로 표기한다.


### TC-11 short 오버플로우 값 입력 (보안)

```bash
java [CP] 65536 20000 30000
```

| 항목 | 내용 |
|---|---|
| 예상 출력 | `에너지 생산량의 범위는 0 ~ 30,000 사이여야 합니다!` |
|  | `현재 입력된 값: 65536` |
| 확인 사항 | `int`로 파싱 후 범위 검사하여 거부된다 |

> `(short) 65536 = 0` 이므로 캐스팅 후 검사하면 0이 통과된다. `int` 상태에서 먼저 검사하여 차단한다.

### TC-13 모두 0 입력 (회귀 테스트)

```bash
java [CP] 0 0 0
```

| 항목 | 내용 |
|---|---|
| 예상 출력 | `-> 총 에너지 사용량은 0입니다.` |
|  | `태양광 0.000000000%, 풍력 0.00000000%, 지열 0.0000000%` |
| 확인 사항 | 총합이 0일 때 비율이 0%로 처리되고 크래시하지 않는다 |

> **회귀 테스트:** 수정 전 코드는 총합이 0일 때 `0.0/0` 나눗셈으로 `NaN`이 발생해
> `new BigDecimal(NaN)`이 `NumberFormatException: Infinite or NaN`을 던지며
> 프로그램이 크래시하는 버그가 있었다. 이 케이스는 해당 버그가 재발하지 않는지
> 확인하기 위한 테스트다.
